# Calibration Guide — Налаштування скорингу

Цей документ призначений для адміністраторів і розробників, які відповідають за підтримку
та калібрування скорингового алгоритму.

---

## 1. Де зберігаються налаштування

**Single source of truth:** `src/lib/scoring/calibration.ts`

Усі магічні числа (пороги архетипів, ваги осей, пороги валідності, коефіцієнти верифікації)
визначені в цьому файлі як експортовані константи. Зміна значення в `calibration.ts`
автоматично поширюється через весь скоринговий рушій без потреби змінювати бізнес-логіку.

Файл експортує чотири константи:

| Константа                    | Призначення                                      |
| ---------------------------- | ------------------------------------------------ |
| `ARCHETYPE_THRESHOLDS`       | Пороги для присвоєння архетипу                   |
| `VALIDITY_THRESHOLDS`        | Пороги для визначення рівня достовірності        |
| `TECH_QUALIFICATION_WEIGHTS` | Бали і коефіцієнти для технічних кваліфікацій    |
| `AXIS_WEIGHTS`               | Бонуси і ваги, що не виводяться з даних запитань |

## 2. `ARCHETYPE_THRESHOLDS` — Пороги архетипів

### Загальні пороги

| Поле                               | Поточне значення | Що означає                                                                                     |
| ---------------------------------- | ---------------- | ---------------------------------------------------------------------------------------------- |
| `weakAxis`                         | 35               | Бал по осі нижче цього значення вважається «слабким»                                           |
| `weakAxisCountForNotSuitable`      | 3                | Кількість слабких осей, при якій присвоюється архетип «Не підходить»                           |
| `strongAxis`                       | 50               | Бал по осі вище цього значення вважається «сильним» (використовується для «Дані недостовірні») |
| `strongAxisCountForDataUnreliable` | 4                | Кількість сильних осей при unreliable validity → архетип «Дані недостовірні»                   |

### Пороги для конкретних архетипів

#### `potentialLeader` — Потенційний лідер

| Поле             | Поточне значення | Умова                 |
| ---------------- | ---------------- | --------------------- |
| `leadership`     | 70               | Лідерство ≥ 70        |
| `responsibility` | 65               | Відповідальність ≥ 65 |
| `initiative`     | 60               | Ініціативність ≥ 60   |

#### `technicalExecutor` — Технічний виконавець

| Поле             | Поточне значення | Умова                                                   |
| ---------------- | ---------------- | ------------------------------------------------------- |
| `technical`      | 60               | Технічна готовність ≥ 60                                |
| `responsibility` | 60               | Відповідальність ≥ 60                                   |
| `leadershipMax`  | 60               | Лідерство < 60 (щоб відрізнити від Потенційного лідера) |

#### `adminCoordinator` — Адміністратор-координатор

| Поле             | Поточне значення | Умова                    |
| ---------------- | ---------------- | ------------------------ |
| `responsibility` | 65               | Відповідальність ≥ 65    |
| `learnability`   | 55               | Навченість ≥ 55          |
| `leadershipMin`  | 40               | Лідерство ≥ 40           |
| `leadershipMax`  | 65               | Лідерство < 65           |
| `technicalMax`   | 40               | Технічна готовність < 40 |

#### `universalPotential` — Універсал з потенціалом

| Поле         | Поточне значення | Умова                              |
| ------------ | ---------------- | ---------------------------------- |
| `minAllAxes` | 55               | Всі осі ≥ 55                       |
| `maxSpread`  | 25               | Різниця між max і min по осях < 25 |

### Як підбирати пороги

- Якщо архетип «Потенційний лідер» присвоюється занадто часто → збільшіть
  `potentialLeader.leadership` або `potentialLeader.responsibility`.
- Якщо «Не підходить» охоплює занадто багато респондентів → збільшіть `weakAxisCountForNotSuitable` до 4
  або підвищіть `weakAxis` до 30.
- Якщо «Дані недостовірні» виникає навіть у сильних профілів → зменшіть `strongAxisCountForDataUnreliable`.

> **Правило:** після будь-якої зміни запустіть `pnpm validate`. Тести в
> `tests/unit/scoring/archetype.test.ts` і `tests/unit/scoring/calibration.test.ts` мають пройти.

## 3. `VALIDITY_THRESHOLDS` — Пороги валідності

### `lie` — Шкала соціальної бажаності

| Поле         | Поточне значення | Що означає                                                         |
| ------------ | ---------------- | ------------------------------------------------------------------ |
| `flagValue`  | 4                | Значення Likert ≥ 4 на lie-питаннях рахується як «прапорець»       |
| `unreliable` | 60               | Lie score ≥ 60 → validity = unreliable                             |
| `reliable`   | 30               | Lie score ≤ 30 → умова для reliable (потрібні також інші критерії) |

### `consistency` — Внутрішня узгодженість

| Поле         | Поточне значення | Що означає                                                                                  |
| ------------ | ---------------- | ------------------------------------------------------------------------------------------- |
| `unreliable` | 60               | Consistency score ≥ 60 → unreliable. Вище значення = більше суперечностей між парами питань |
| `reliable`   | 30               | Consistency score ≤ 30 → умова для reliable                                                 |

### `attention` — Перевірки уважності

| Поле         | Поточне значення | Що означає                                                    |
| ------------ | ---------------- | ------------------------------------------------------------- |
| `unreliable` | 50               | Attention score < 50% пройдених attention checks → unreliable |
| `reliable`   | 75               | Attention score ≥ 75% → умова для reliable                    |

> Нижчий поріг `attention.unreliable` означає суворіший контроль (більше респондентів
> отримуватимуть unreliable). Підвищення порогу послаблює вимогу.

### `speed` — Швидкість відповідей

| Поле                 | Поточне значення | Що означає                                                         |
| -------------------- | ---------------- | ------------------------------------------------------------------ |
| `minMedianMs`        | 3000             | Медіана часу відповіді < 3000 мс у Блоках 4–5 → speed flag = true  |
| `uniformityFraction` | 0.8              | ≥ 80% однакових Likert-відповідей → speed flag = true              |
| `minLikertCount`     | 5                | Мінімальна кількість Likert-відповідей перед перевіркою uniformity |

### Загальна логіка overall validity

```
unreliable: lie ≥ 60 OR consistency ≥ 60 OR attention < 50% OR speed flag = true
reliable:   lie ≤ 30 AND consistency ≤ 30 AND attention ≥ 75% AND speed flag = false
questionable: все решта
```

## 4. `TECH_QUALIFICATION_WEIGHTS` — Кваліфікаційні ваги

### `basePoints` — Базові бали за кваліфікацію

| Кваліфікація      | Поточне значення | Опис                               |
| ----------------- | ---------------- | ---------------------------------- |
| `demining`        | 25               | Сапер / EOD-досвід                 |
| `drone-piloting`  | 25               | Пілотування дронів (будь-який тип) |
| `radar-radiotech` | 25               | Знання РЛС / радіотехніки          |

### `driving` — Бали за водійські категорії

| Поле    | Поточне значення | Умова                                    |
| ------- | ---------------- | ---------------------------------------- |
| `heavy` | 10               | Категорія C або D (вантажівка / автобус) |
| `other` | 7                | Будь-яка інша категорія (B, A, E)        |

### Нормалізація

| Поле     | Поточне значення | Опис                                                           |
| -------- | ---------------- | -------------------------------------------------------------- |
| `maxRaw` | 85               | Знаменник для нормалізації сирої суми балів до діапазону 0–100 |

Якщо додається нова кваліфікація з базовими балами, `maxRaw` треба збільшити на відповідне
значення, щоб максимально можливий бал залишався досяжним.

### `verification` — Коефіцієнти верифікації

| Поле            | Значення      | Що означає                                                            |
| --------------- | ------------- | --------------------------------------------------------------------- |
| `verifiedRatio` | 2/3 (≈ 0.667) | ≥ 2/3 правильних відповідей → множник 1.0 (Верифіковано)              |
| `partialRatio`  | 1/3 (≈ 0.333) | ≥ 1/3 правильних відповідей → множник 0.5 (Часткова верифікація)      |
| —               | < 1/3         | → множник 0.0 (Не верифіковано, прапорець «ймовірно неправдиві дані») |

## 5. Додавання нової технічної кваліфікації

Щоб додати нову кваліфікацію (наприклад, `medical-training`):

1. Додайте її до `TECH_QUALIFICATION_WEIGHTS.basePoints` у `calibration.ts`:

   ```ts
   'medical-training': 20,
   ```

2. Оновіть `maxRaw` у тому самому файлі (додайте 20 до поточного значення):

   ```ts
   maxRaw: 105,
   ```

3. Додайте верифікаційні запитання до `src/data/questions/verification.json`.

4. Додайте питання-декларацію (чек-бокс) до `src/data/questions/qualifications.json`.

5. Оновіть `src/lib/scoring/technical-verification.ts` — додайте нові ID до
   `VERIFICATION_QUESTIONS` і відповідний declaration question ID до `DECLARATION_QUESTION_IDS`.

6. Запустіть `pnpm validate` — всі перевірки мають пройти.

## 6. Додавання/видалення питань

> **Увага:** ID запитань використовуються безпосередньо в файлах `src/lib/scoring/axes/*.ts`.
> Видалення або перейменування ID може призвести до тихих помилок скорингу.

### Кроки для додавання запитання

1. Відредагуйте відповідний JSON-файл у `src/data/questions/`.
2. Додайте посилання на новий ID у відповідний файл осі в `src/lib/scoring/axes/`, якщо
   питання впливає на скоринг.
3. Оновіть `CLAUDE.md`, якщо змінилася структура блоків (кількість питань, нові блоки).
4. Запустіть `pnpm validate` — всі перевірки мають пройти.

### Кроки для видалення запитання

1. Перевірте, чи не використовується ID у файлах `src/lib/scoring/axes/*.ts` та
   `src/lib/scoring/technical-verification.ts`.
2. Якщо використовується — видаліть або замініть посилання у відповідному файлі.
3. Видаліть запитання з JSON-файлу.
4. Запустіть `pnpm validate`.

## 7. Процес калібрування на реальних даних

Початкові пороги в `calibration.ts` задані емпірично. Після пілотного використання
рекомендується провести калібрування на реальних даних:

### Покрокова процедура

1. **Зберіть вибірку.** Необхідно N ≥ 50 профілів із відомими результатами (люди,
   чия реальна компетентність підтверджена незалежно — через рекомендації, виконання
   задач, оцінки керівників тощо).

2. **Порівняйте архетипи з фактичними результатами.** Для кожного профілю:
   - Який архетип присвоїв інструмент?
   - Яким виявився реальний результат?
   - Зафіксуйте False Positives (сильний архетип → слабкий кандидат) та
     False Negatives (слабкий архетип → сильний кандидат).

3. **Відрегулюйте пороги.** Мета — мінімізувати суму похибок I і II типу для
   пріоритетних ролей. Наприклад:
   - Якщо «Потенційний лідер» занадто часто виявляється помилковим → збільшіть пороги.
   - Якщо реальні лідери часто потрапляють до «Виконавець базового рівня» → знизьте пороги.

4. **Запустіть `pnpm validate`** після кожної зміни. Переконайтеся, що всі тести проходять.

5. **Задокументуйте зміни.** Зафіксуйте в коментарях до commit або у changelog:
   - Розмір вибірки
   - Дату калібрування
   - Які пороги змінено і чому

### Тести для перевірки після змін

Після будь-яких змін у `calibration.ts` або у файлах запитань — обов'язково:

```bash
pnpm test               # unit тести скорингу
pnpm test:e2e           # повний flow e2e
pnpm typecheck          # TypeScript
```

Особлива увага до:

- `tests/unit/scoring/calibration.test.ts` — перевірка структури і граничних значень констант
- `tests/unit/scoring/archetype.test.ts` — кейси присвоєння архетипів
- `tests/unit/scoring/axes-scoring.test.ts` — скоринг по кожній осі

## 8. `AXIS_WEIGHTS` — Бонуси по осях

| Поле                        | Поточне значення | Що означає                                                                                                                     |
| --------------------------- | ---------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `leadership.largeTeamBonus` | 5                | Бонусні бали до лідерства, якщо респондент заявив досвід керування > 10 людьми. Максимальний бал по осі залишається 100 (cap). |

Ця константа відокремлює бонуси, які не можна вивести безпосередньо з відповідей на
конкретні запитання, від ваг, що обраховуються динамічно.
